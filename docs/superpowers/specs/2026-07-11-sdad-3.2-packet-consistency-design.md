# SDAD 3.2 Packet Consistency Design

Status: Approved design, awaiting written-spec review
Date: 2026-07-11
Target release: SDAD v3.2.0
Extends: `2026-07-10-sdad-doctor-design.md`

## 1. Purpose

SDAD 3.1 made the state-to-INDEX route compact and diagnosable, but a mature
project migration exposed consistency gaps that structural validation alone
cannot prevent:

- a long-running umbrella objective can be placed in `active_packet` even
  though the field is intended to name one bounded checkpoint;
- an active packet can change while its validation claims still describe the
  previous packet;
- INDEX or a continuity document can point at a handoff for a different packet;
- completed work can remain in active TODO or finding sections;
- a downstream project can invoke a different released Doctor than the one its
  validation contract expected;
- platform-specific command syntax can be mistaken for portable validation.

This design turns the generalizable parts of those failures into explicit,
deterministic contracts. It does not add natural-language semantic guessing or
make Doctor a product-correctness oracle.

The design applies to stateful Standard and Full SDAD. One-shot and Mini retain
their file-free or one-file contracts. A deliberately stateful Mini project
continues on state v1, but it does not migrate to v2.

## 2. Goals

1. Preserve v1 state interpretation, findings, severities, human output, exit
   behavior, and the report-schema-v1 JSON shape for existing invocations.
2. Make the active packet the explicit identity spine for validation, handoff,
   and active TODO/review records in state v2.
3. Detect stale packet associations without comparing natural-language prose.
4. Keep handoff context out of the fixed startup budget unless the task is a
   resume or handoff.
5. Prevent active-ledger cleanup rules from interrupting normal in-progress
   checklist use.
6. Fail closed when a canonical command invokes the wrong released Doctor.
7. Give mature projects an explicit migration-preview procedure before any
   control files are rewritten.
8. Preserve Doctor's read-only, no-command-execution, no-network boundary.

## 3. Non-Goals

The following are not part of v3.2.0:

- semantic similarity or model-based comparison of `objective` and `proves`;
- `parent_packet_id`, a program object, or an `objective_chain` in startup state;
- nested `routed_docs.startup` and `routed_docs.on_demand` schema;
- a required or automatically loaded current handoff;
- arbitrary shell parsing or `--check-command-paths`;
- validation `platform`, `runner`, or `shell` metadata;
- a mutating migration command, YAML rewriter, or `sdad migrate --apply`;
- a standalone Doctor package, zipapp, package-manager install, or release asset;
- a Doctor runtime lock or self-hashing manifest;
- validation-command execution, auto-fix, Git inspection, telemetry, or network
  access;
- changes to One-shot or Mini persistence rules;
- changes to the approved README infographic;
- independent or divergent edits to the README Copy-Paste Start Prompt and
  no-clone Option 1. Their shared canonical text will gain the minimum v3.2
  adoption rules and release-owned pins, but must remain byte-identical and the
  README block must remain expanded;
- changes to the field-report project used to motivate this design.

If same-version dirty or forked Doctor rejection becomes an observed need, it
requires a separate distribution and provenance design. Exact released-version
selection is sufficient for the currently observed problem.

## 4. Existing Contracts To Preserve

- AI-complete/evidence-ready and owner-accepted remain distinct.
- `owner_gates` contains only gates capable of stopping the current packet.
- Doctor never executes the commands declared under `validation`.
- Doctor findings prove only the named control-plane checks.
- Project content remains untrusted input confined to the accepted project root.
- `routed_docs` and declared paths remain normalized repository-relative POSIX
  paths.
- Unknown or unsupported state must never produce a false-clean report.
- The fixed startup control plane remains at or below 12,000 characters.
- The active source, tests, and runtime evidence outrank summaries and handoffs.
- The canonical adapter kernel remains compact and rendered adapters remain
  deterministic copies.
- v1 projects do not receive new v2-only findings merely because the Doctor was
  upgraded or because report schema 2 was explicitly selected.

## 5. Terminology And Version Domains

Five version domains must remain separate:

| Domain | v3.2 value | Meaning |
|---|---:|---|
| SDAD release | `3.2.0` | Documentation, templates, Doctor, and release bundle |
| Doctor executable | `3.2.0` | Exact CLI behavior required by `--require-version` |
| State schema | `1` or `2` | Meaning of `sdad-state.yaml` |
| Doctor JSON report schema | `1` or `2` | Existing-invocation compatibility versus the opt-in/v2 output shape |
| Install-source manifest schema | `1` | Existing adapter-source manifest structure |

Implementation names must make the domains explicit, for example:

```text
DOCTOR_VERSION = "3.2.0"
REPORT_SCHEMA_VERSION = 2
LEGACY_REPORT_SCHEMA_VERSION = 1
SUPPORTED_STATE_VERSIONS = {1, 2}
```

The generic name `SCHEMA_VERSION` must not be used for more than one domain.

## 6. Architecture

The existing thin CLI, deterministic engine, confined project view, and fixed
check pipeline remain. Version dispatch and packet identity are added without a
plug-in system.

```text
CLI arguments
    |
    +--> exact Doctor version gate --------------------------+
    |        mismatch -> diagnostic_error / exit 2           |
    v                                                       |
confined ProjectView                                        |
    |                                                       |
    v                                                       |
parse sdad-state.yaml                                       |
    |                                                       |
    +--> missing version -> effective v1 + existing warning |
    +--> supported v1  -> v1 rules only                     |
    +--> supported v2  -> v2 rules + identity spine         |
    +--> unsupported   -> state_schema only; skip dependents|
    v                                                       |
fixed Doctor checks                                         |
    |                                                       |
    +--> state schema and freshness                         |
    +--> declared path integrity                            |
    +--> validation ownership and evidence-contract shape   |
    +--> owner gates                                        |
    +--> TODO/findings/handoff coherence                    |
    v                                                       |
human report or version-selected JSON report <--------------+
```

No new dynamic check registry is introduced. Existing checks are extended in
their current responsibility boundaries. A new module is justified only if the
implementation plan proves that handoff parsing cannot remain isolated and
testable inside the existing review/continuity boundary.

## 7. State Schema v2

### 7.1 Canonical Shape

```yaml
version: 2
updated: YYYY-MM-DD

# state v2: standard | full
scale: standard
# low | medium | high
intensity: medium
# 0 = ask-first, 1 = unit, 2 = work packet, 3 = session, 4 = owner-gated risk
autonomy: 2

active_spec: SPEC/SPEC-COMPLETE.md
active_packet:
  id: WP-HTTPS-EDGE
  objective: Produce one bounded, observable, evidence-ready outcome.
  status: in_progress

# Must equal active_packet.id exactly. Re-review every command and proves value
# before changing this association during a packet transition.
validation_for: WP-HTTPS-EDGE

# Optional. Uncomment only when a current resume document actually exists.
# current_handoff: docs/sdad/handoffs/YYYY-MM-DD-topic.md

owner_gates: []

validation:
  - command: python scripts/validate.py
    proves: The current packet acceptance checks pass in the configured environment.

# Immediately selectable current-packet paths; read only what intent selects.
routed_docs:
  - docs/TODO-Open-Items.md
  - review-findings.md
```

`current_handoff` is omitted when no current continuity document exists. It does
not use `none`, `null`, an empty string, or an empty collection sentinel.

### 7.2 Required And Optional Keys

The parser must separate known keys from required keys by state version.

```text
REQUIRED_TOP_LEVEL_KEYS_BY_VERSION
KNOWN_TOP_LEVEL_KEYS_BY_VERSION
REQUIRED_PACKET_KEYS_BY_VERSION
KNOWN_PACKET_KEYS_BY_VERSION
```

v1 retains its existing required and known keys. v2 adds required
`validation_for` and optional `current_handoff`. v2 does not add an
`active_packet.parent` field.

The v2 schema owns malformed identity fields so dependent checks do not emit
duplicates:

| Defect | Finding ID |
|---|---|
| missing `validation_for` | `state.schema.missing-key` |
| non-scalar `validation_for` | `state.schema.wrong-kind` |
| blank or grammar-invalid `validation_for` | `state.schema.unsupported-value` |
| blank `active_packet.id` | existing `state.packet.blank-field` |
| grammar-invalid `active_packet.id` | `state.schema.unsupported-value` |
| non-scalar `current_handoff` | `state.schema.wrong-kind` |
| scalar blank or non-normalized `current_handoff` | `path.invalid` |

State v2 permits only `scale: standard` or `scale: full`. One-shot remains
file-free and Mini remains a one-file control mode; creating v2 state for either
is `state.schema.unsupported-value`. A deliberately stateful Mini stays on v1.
State v1 retains its existing enum behavior for compatibility.

A v1 document containing a v2-only key receives the existing
`state.schema.unknown-key` behavior. It must not be silently treated as a v2
document.

### 7.3 Effective-Version Dispatch

- Missing `version` means effective v1 and preserves the existing missing-version
  warning.
- Declared `version: 1` uses only v1 requirements and diagnostics.
- Declared `version: 2` uses v2 requirements and diagnostics.
- Any other declared version emits `state.schema.unsupported-version`.
- Unsupported versions run only state-schema diagnostics. Path, packet, gate,
  and review checks are skipped because their semantics are unknown.
- Parser failure continues to skip every dependent check.

The state result must expose the effective supported version to checks without
requiring each check to reparse a raw scalar.

### 7.4 Canonical Packet ID

State v2 packet IDs use this grammar:

```regex
^[A-Za-z0-9][A-Za-z0-9._-]{0,63}$
```

The same grammar applies to `validation_for`, handoff packet markers, and v2
active-ledger packet markers. v1 IDs retain their existing compatibility
behavior.

Invalid v2 IDs use the existing `state.schema.unsupported-value` ID as errors.
Exact matching is case-sensitive. Doctor never trims one side of an identity
comparison and then compares it with an untrimmed value.

### 7.5 Umbrella Objectives

`active_packet` always names one executable leaf checkpoint. A program, roadmap,
post-MVP track, or umbrella objective remains in the active SPEC or an on-demand
work-packet ledger.

After a child packet ends, state must either:

- keep the completed child until the next leaf is selected; or
- advance directly to the next leaf packet.

It must not restore an umbrella objective as the executable active packet.

No parent identifier is added in v3.2 because no canonical packet registry can
prove parent existence, spelling, or cycles. An unverifiable parent would create
a second apparent authority without preventing scope drift.

This is a procedural agent/owner contract. Doctor can verify explicit IDs and
their references, but it cannot determine from objective prose whether a packet
is truly a leaf. A semantically broad umbrella may therefore still receive a
clean structural report; migration and transition review are the mitigation.

## 8. Validation Ownership

`validation_for` is a referential-integrity field, not semantic proof.

For valid v2 state:

```text
validation_for == active_packet.id
```

A mismatch emits `validation.packet-mismatch` as an error. Missing or malformed
fields remain schema-owned so the same defect does not receive duplicate
findings.

Packet transition is one coherence transaction, even though the underlying file
writes are not atomic:

1. Choose one new executable leaf packet ID and outcome without changing state
   yet.
2. Rehome the previous packet's closed, deferred, and still-open active-ledger
   records. History moves to `## Recently Closed` or an archive; future work
   moves to its non-active section; deliberately carried work is relinked to the
   selected next packet.
3. Inspect every validation entry; remove previous-packet-only checks, rewrite
   stale `proves` claims, add the smallest task-specific acceptance check, and
   retain broad regression checks only with bounded claims.
4. Update `updated`, `active_spec`, `active_packet.id`, objective, initial status,
   `validation_for`, owner gates, validation, and routes as one declaration
   bundle. `validation_for` changes only after step 3.
5. Remove the old `current_handoff` or replace it with a new handoff for the new
   packet. Never carry the old pointer across mechanically.
6. Ensure every active TODO/finding record uses the new packet identity, then run
   canonical Doctor strict mode on the coherent declaration.
7. Perform the work and run project validation separately. Record the evidence.
8. Advance to an evidence checkpoint only after its evidence exists, then rerun
   Doctor strict mode because status-sensitive severity may have changed.

Agents should finish steps 1-6 before treating an intermediate Doctor failure as
a project defect. This avoids needless reruns without allowing a transient file
state to be reported as coherent.

A clean Doctor report means the validation block declares the current packet as
its owner and has a structurally usable contract. It does not mean the commands
are executable, have run, cover the objective, or prove product correctness.

## 9. Optional Current Handoff

### 9.1 Single Canonical Pointer

When a current session handoff exists, v2 state must declare
`current_handoff`. That state field is the only canonical current-handoff path.
INDEX routes a resume operation to the state pointer; it does not duplicate a
project-specific current path.

The pointer is optional because handoffs are created only when work pauses,
changes hands, remains blocked or partial, or would be expensive to reconstruct.

The always-available `save-state.md` template remains a generic compact
continuity summary; its mere existence does not require `current_handoff`. The
field designates a timestamped or otherwise packet-specific handoff selected as
the current resume checkpoint.

It means "the latest resume checkpoint for the active packet," not live state.
Within one packet it remains valid until replaced by a newer checkpoint. At a
packet transition it must be removed or replaced. Doctor verifies packet
identity, not temporal freshness or factual completeness inside a same-packet
handoff.

### 9.2 Path Contract

When present, `current_handoff` must be:

- a normalized repository-relative POSIX path;
- confined to the accepted project root after link resolution;
- an existing regular file;
- readable as UTF-8;
- within the existing control-document byte limit.

Existing path finding IDs and size limits are reused except for an oversized
canonical handoff. `path.too-large` is only a warning for ordinary routed
documents, so a `current_handoff` that exceeds the limit emits the dedicated
error `handoff.path.too-large`. If path inspection or content decoding fails,
marker checks are suppressed to avoid cascade findings. The remaining path and
encoding failures are already errors.

### 9.3 Marker Contract

The first exact `## 1. Session Identity` section contains exactly one canonical
line:

```markdown
- Active packet: [packet:WP-HTTPS-EDGE]
```

Only that exact field is authoritative. Packet markers in examples, completed
work, decisions, quoted text, or reactivation prompts do not satisfy the check.

Within the selected section, candidate lines begin exactly `- Active packet:`.
Precedence is deterministic: zero candidates is missing; more than one is
duplicate only; one candidate that does not match the full canonical line and ID
grammar is invalid; and one valid marker unequal to `active_packet.id` is a
mismatch. A candidate receives only the highest-precedence finding, avoiding a
cascade.

If the exact Session Identity section itself is absent, Doctor emits
`handoff.structure.missing-marker` with section evidence; there is no second
missing-section finding. Later duplicate Session Identity sections are ignored
and cannot satisfy the first-section contract.

Stable finding IDs:

- `handoff.structure.missing-marker`;
- `handoff.structure.duplicate-marker`;
- `handoff.structure.invalid-marker`;
- `handoff.packet-mismatch`.

### 9.4 Context Budget

Declaring a handoff does not add it to the default agent startup read. The agent
loads it only for resume/handoff intent or when continuity is otherwise needed.
Doctor inspects the declared file for coherence without placing its content in
the model context.

The current measured fixed startup surface is 9,892 characters. The current
handoff template is 3,245 characters; loading it by default would raise the
surface to 13,137 and violate the 12,000-character contract. The implementation
must preserve conditional loading.

The projected v2 state adds roughly 237 characters. Together with one compact
adapter Fast Start clarification, the fixed default surface is expected to stay
below roughly 10,250 characters. A resume operation may intentionally exceed the
fixed-startup budget after it selects the 3,245-character handoff; that
conditional context is not part of the always-read surface and must still use
targeted reads. Implementation remeasures the exact rendered value.

### 9.5 INDEX Source Guard

For state v2, Doctor inspects fixed canonical path `docs/INDEX.md` exactly once,
reusing a cached read if another check already selected it. Existing path,
encoding, and size findings apply.

The canonical v2 INDEX contains this exact line in its first `## Active Catalog`
section:

```markdown
- Current handoff: use `../sdad-state.yaml#current_handoff` when declared.
```

On a readable v2 INDEX, a missing, duplicate, or different `- Current handoff:`
line emits warning `index.current-handoff-source`. This narrow check catches the
observed stale project-specific path without trying to interpret arbitrary
Markdown links. INDEX remains routing authority, but it never becomes a second
current-handoff pointer.

If INDEX cannot be read or is oversized, existing path findings apply and this
source check is suppressed. Consequently this warning is a deterministic guard
for a readable canonical INDEX, while migration preview remains the safeguard
for older custom INDEX layouts. The design does not claim to discover every
natural-language reference to an old handoff.

## 10. Active-Ledger Hygiene

Doctor continues to inspect only exact active sections:

- `review-findings.md` -> `## Active Findings`;
- `docs/TODO-Open-Items.md` -> `## Active Work` and
  `## Release / Production Readiness`.

`## Recently Closed`, future/deferred sections, and archives remain outside this
check.

### 10.1 Exact v2 Record Grammar

State v1 retains the existing permissive marker behavior. In state v2, every
record-like bullet in an exact active section must use one of these complete
forms, where `ID` follows the v2 packet grammar and description is non-empty:

```text
review open:    - [Critical|High|Medium|Low] [packet:ID] description
review open:    - [packet:ID] description
review closed:  - [x|X] [Critical|High|Medium|Low] [packet:ID] description
review closed:  - [x|X] [packet:ID] description
TODO open:      - [ ] [packet:ID] description
TODO closed:    - [x|X] [packet:ID] description
```

The notation above represents alternatives; literal lines contain one severity
or one checkbox value, for example `- [x] [High] [packet:WP-7] Fixed overflow`.
Spaces and token order are exact. Packet values are not trimmed before grammar
or identity comparison. `None currently tracked.` and non-bullet explanatory
prose remain sentinels, not records. Other `- ` bullets inside an active section
are malformed records rather than prose.

For open records, no packet-looking token is
`ledger.open-item-missing-marker`. Invalid packet delimiters or an ID outside the
v2 grammar are `ledger.open-item-invalid-marker`. A valid marker in a line that
otherwise violates record grammar--including unsupported severity, token order,
spacing, or empty description--is `ledger.open-item-malformed-record`. A fully
valid record whose marker names another packet is
`ledger.open-item-packet-mismatch`.

### 10.2 Open-Record Identity

For this section, terminal means exactly `owner_accepted` or
`production_ready`; non-terminal means every other valid packet status,
including `blocked` and `deferred`.

For v2, an open record linked to the current packet retains the existing
checkpoint-sensitive `packet.open-finding`, `packet.open-critical-finding`, and
`packet.open-todo` behavior. Identity defects use these stable IDs:

| Open active record | Finding ID | Non-terminal | `owner_accepted` / `production_ready` |
|---|---|---:|---:|
| packet marker missing | `ledger.open-item-missing-marker` | Warning | Error |
| packet marker malformed or ID grammar invalid | `ledger.open-item-invalid-marker` | Warning | Error |
| marker valid but record grammar malformed | `ledger.open-item-malformed-record` | Warning | Error |
| valid marker names another packet | `ledger.open-item-packet-mismatch` | Warning | Error |

This makes old-packet open work observable during a packet switch. Work that is
not active belongs in `## Future / Deferred`; deliberately carried work must be
relinked as part of the transition transaction.

### 10.3 Closed-Record Hygiene

Review findings and TODOs have different lifecycle semantics:

| Closed record left in an active section | When no finding is emitted | Warning | Error |
|---|---|---|---|
| Review finding | Never | every non-terminal status | `owner_accepted`, `production_ready` |
| Full valid TODO linked to current packet | only `in_progress` | all other non-terminal statuses | terminal status |
| TODO malformed or marker missing, invalid, or different | Never | every non-terminal status | terminal status |

Stable IDs are `ledger.closed-review-in-active-section` and
`ledger.closed-todo-in-active-section`. Within each ID, severity depends only on
terminal versus non-terminal packet status; linkage changes whether the TODO
finding is emitted during `in_progress`, not the severity of an emitted finding.
This fits the existing central validator without adding hidden linkage state to
the public `Finding` model.

The TODO template gains `## Recently Closed` as the normal short-term destination.
Closed review findings already move to their existing `## Recently Closed`
section before `ai_complete`, save-state, or handoff. Old closed history may then
move to archive. These destination sections remain outside active checks.

Schema or packet-ID failure suppresses dependent ledger identity findings. An
open line receives only one missing/invalid/malformed/mismatch classification. A
closed line receives only its closed-review or closed-TODO finding; malformed
closed TODOs are not eligible for the `in_progress` quiet case. Thus one record
does not produce a cascade.

## 11. Routed Document Semantics

`routed_docs` remains a flat scalar path list.

Its precise meaning is:

> the minimum set of current-packet paths that may be selected immediately after
> the adapter, state, INDEX, and source/test route. INDEX and current intent
> select which listed document, heading, active section, or targeted match to
> read. Membership never means load every file in full. INDEX may route
> additional on-demand policy or evidence for the current intent or gate.

Evidence matrices, claim registries, large packet ledgers, archives, and old
handoffs do not belong in this list unless they can change the current packet's
next decision. Active headings and bounded reads remain preferred even for a
listed path. No nested startup/on-demand route schema is added.

## 12. Doctor Version Guard

### 12.1 CLI Contract

```text
python <SDAD_CHECKOUT>/scripts/sdad.py --version

python <SDAD_CHECKOUT>/scripts/sdad.py doctor [PROJECT_ROOT] \
  --require-version 3.2.0 [--json] [--strict]
```

Rules:

- `--version` is a top-level standalone operation. It prints exactly `3.2.0\n`
  to stdout and exits `0` without constructing a project view or reading a
  project. Combining it with a subcommand or other argument is invalid.
- `--require-version` is Doctor-only, may appear before or after `PROJECT_ROOT`,
  and may occur exactly once. Split form `--require-version 3.2.0` and equals
  form `--require-version=3.2.0` are both accepted and each counts as one
  occurrence. `--` ends option parsing as usual.
- The accepted value is core SemVer `X.Y.Z` matching
  `^(0|[1-9][0-9]*)\.(0|[1-9][0-9]*)\.(0|[1-9][0-9]*)$`. Ranges, leading zeroes,
  whitespace, prerelease, and build metadata are unsupported.
- Malformed or duplicate requirements use existing `invalid_invocation` and
  exit `2`. A syntactically valid value unequal to `DOCTOR_VERSION` uses new
  diagnostic-error kind `version_mismatch` and exit `2`.
- Syntax and equality are checked before `FilesystemProjectView` construction
  or any project filesystem access. Therefore mismatch wins over a nonexistent
  `PROJECT_ROOT`; a matching value followed by a bad root uses existing
  `unusable_root`.
- Doctor help exits `0` without version comparison or project access. Explicit
  help is human output even if `--json` also appears. All non-help JSON failures
  still emit exactly one JSON document.
- A v3.1 Doctor also fails closed because it does not recognize the new option
  and already returns invocation exit `2` for unknown arguments.
- Canonical downstream Doctor commands include the exact requirement. Running
  Doctor manually without the flag has no version-lane protection.

Doctor version changes only when CLI, state interpretation, check behavior, or
report behavior changes. A documentation-only SDAD patch may retain the same
Doctor version. Adopting a changed Doctor requires rotating downstream exact
requirements; this deliberate maintenance cost prevents silent version drift.

### 12.2 JSON Report Selection

Existing JSON calls remain compatible:

1. An explicit `--require-version` token recognized before `--` selects report schema 2 for any
   non-help JSON document, including malformed, duplicate, or mismatched uses.
2. Without that flag, establishing effective state v2 selects report schema 2
   for both completed diagnosis and any later exit-2 failure.
3. Without that flag, effective-v1, missing-state, unsupported-version, and
   existing invocation/error lanes retain the exact report-schema-v1 shape.

Thus existing v1 clients do not migrate merely by upgrading Doctor. A caller
opts into schema 2 by using the new guard, and a state-v2 project necessarily
uses schema 2. Selecting schema 2 never enables v2-only checks for v1 state.

A completed schema-2 document has these exact top-level keys in this order:

```json
{
  "schema_version": 2,
  "doctor_version": "3.2.0",
  "state_version": 2,
  "root": "C:/accepted/project",
  "strict": false,
  "summary": {"errors": 0, "warnings": 0},
  "checks": {"run": [], "skipped": []},
  "findings": []
}
```

The existing finding object shape and summary/check shapes do not change. A
missing `version` that takes the legacy lane reports effective
`state_version: 1`; missing state and unsupported semantics report `null`.

A schema-2 exit-2 document has the same first eight top-level keys, with empty
summary/check/finding values as in schema 1, followed by:

```json
"diagnostic_error": {"kind": "version_mismatch", "message": "..."}
```

`state_version` is `null` when no supported state was established. A no-guard
failure before effective v2 is established remains schema 1; a failure in a
later v2 check is schema 2 with `state_version: 2`. `root` is
`null` when failure occurs before project-root acceptance, including malformed
or mismatched version gates; after root acceptance it is the existing normalized
root string. Existing schema-1 key order, values, and four pre-v3.2 diagnostic
error kinds remain byte-shape compatible for calls that stay in schema 1.

The engine centrally enforces separate allowed-finding sets for effective v1
and v2. Emitting any v2-only finding for v1 is an internal diagnostic failure,
not a reportable project finding.

### 12.3 Claim Boundary

The guard prevents accidental use of a different released Doctor version. It
does not detect:

- a dirty or forked Doctor that retains the same version string;
- a compromised runtime and project changed together;
- a moving project state during diagnosis;
- host-date changes that affect freshness warnings;
- different absolute root paths in JSON.

Documentation calls this an exact Doctor version guard, not bitwise result
reproducibility or authenticated provenance.

## 13. Cross-Platform Validation Commands

State v2 does not add `platform`, `runner`, or `shell` fields. Doctor does not
execute commands and therefore cannot establish that a platform-specific entry
provides complete cross-platform coverage.

Preferred order:

1. expose one repository-root, shell-neutral validation wrapper such as
   `python scripts/validate.py`;
2. let the wrapper or CI select platform-specific details;
3. use the CI matrix for multi-OS guarantees;
4. route long operator-specific commands through an operations document;
5. keep each state `proves` claim limited to the environment actually checked.

Doctor does not parse arbitrary shell commands. Executable discovery cannot
prove version, permissions, daemon availability, dependencies, arguments, or
success, and scanning PATH may cross the existing project-root boundary. A
future capability-lint design must use explicit structured requirements rather
than infer them from command text.

Doctor remains checkout-only, so locating `<SDAD_CHECKOUT>` is still an operator
or project-wrapper responsibility. A portable project wrapper may resolve a
documented environment variable and invoke Doctor with `--strict
--require-version 3.2.0`; the exact guard detects a wrong released checkout but
does not make the checkout location self-configuring. No PowerShell-only
`$env:...` command is presented as cross-platform validation.

## 14. Migration Preview Protocol

v3.2 does not add `sdad migrate --plan`. The key migration decisions are
semantic and cannot safely be generated by the current parser:

- whether a legacy target is an umbrella or an executable leaf;
- whether each validation claim belongs to the new leaf;
- which current continuity document, if any, remains authoritative;
- which historical material can leave the active startup route.

Instead, existing-project adoption in the SDAD start skill and migration guide
uses a mandatory read-only preview before writes.

### 14.1 Mature Pre-v3 Adoption Preview

The agent reports:

1. worktree status, existing control files, byte/line sizes, and current
   authority, including dirty or untracked user material that must be preserved;
2. a pre-change Doctor result when the existing state is diagnosable, otherwise
   a read-only structural snapshot and the reason Doctor is not applicable;
3. whether the project is One-shot/stateless Mini, deliberately stateful Mini,
   v1 Standard/Full, or a mature pre-v3 layout, and therefore whether v2
   migration applies;
4. active current records versus history/archive candidates;
5. exact preservation strategy for history before compaction;
6. umbrella objectives versus the first executable leaf packet;
7. proposed validation commands and the bounded claim of each;
8. proposed immediately selectable current-packet routes and their targeted
   read strategy;
9. whether a current handoff exists and should remain current;
10. owner-controlled decisions or evidence gates that cannot be inferred;
11. proposed state, INDEX, ledger, and handoff changes without applying them;
12. the post-change comparison plan: Doctor strict output and actual project
    validation are captured separately and compared with the baseline.

The agent may continue after the preview when the existing request authorizes
adoption and no blocking ambiguity remains. It asks the owner only when choosing
a leaf, authority, data boundary, or gate would materially change intent.

### 14.2 v1 To v2 Checklist

If the declared scale is One-shot or Mini, migration is not applicable; retain
the scale's compact contract instead of creating v2 state. A deliberately
stateful Mini remains on state v1 unless a separate scale escalation is approved.

1. Confirm `active_packet` is one executable leaf, not an umbrella.
2. Normalize its ID to the v2 grammar.
3. Re-evaluate every validation command and `proves` claim.
4. Add `validation_for` equal to the packet ID.
5. Add `## Recently Closed` to TODO when absent, convert every active TODO/review
   record to the exact v2 grammar, and reconcile its marker with the packet.
6. Replace any project-specific INDEX current-handoff path with the exact
   canonical state-source line.
7. Add `current_handoff` only when a real current resume document exists, and
   add its canonical packet marker when it does.
8. Change the state version to `2` only after steps 1-7 form one coherent set.
9. Run Doctor with `--strict --require-version 3.2.0`.
10. Run project validation commands separately; Doctor does not run them.
11. Compare the post-change Doctor/project evidence with the recorded baseline
    and explain intentional differences.

v1 remains supported, so migration is opt-in rather than a strict-mode surprise.

## 15. Diagnostic And Error Handling Rules

- Every new finding ID belongs to the state-v2 finding allowlist; the v1 and v2
  allowed sets are centrally distinct regardless of selected JSON shape.
- Conditional severity remains centrally enforced by finding ID and packet
  status. Linkage changes finding presence, not the allowed severity of an
  emitted ID at the same status.
- Schema defects suppress dependent identity findings.
- Path failures suppress handoff marker findings for the same declaration;
  oversized canonical handoff remains a fixed error.
- Unsupported state versions skip version-dependent checks.
- Duplicate occurrences are deduplicated by ID, path, and line as today.
- Human output states observed evidence and one actionable remediation.
- JSON output remains exactly one document on completed diagnosis and invocation
  failure.
- `--strict` changes exit policy only; it does not reclassify warnings.
- Version mismatch uses exit `2`; coherent diagnosis with policy findings uses
  exit `1` as today.

## 16. Test Design

### 16.1 State Contract

- all existing v1 fixtures and violation strings remain unchanged;
- version missing still selects effective v1 with the current warning;
- a valid v2 state produces a v2 snapshot;
- v2 requires `validation_for` and accepts optional `current_handoff`;
- v2 rejects One-shot or Mini scale while v1 preserves its existing scale enum;
- v1 treats v2-only fields with existing unknown-key behavior;
- unsupported versions do not run v1 or v2 dependent checks;
- v2 packet IDs cover minimum/maximum length and every invalid character class;
- canonical template validation surfaces every v2 error rather than only legacy
  messages.

### 16.2 Validation Ownership

- exact match passes;
- case, whitespace, old packet, blank, missing, wrong-kind, and invalid-ID cases
  fail at the correct layer;
- malformed state suppresses mismatch cascades;
- a matching ID does not create a claim of semantic objective coverage.

### 16.3 Current Handoff

- absence passes;
- normalized in-root UTF-8 handoff with one canonical marker passes;
- missing, outside-root link, directory, unreadable, oversized, and invalid UTF-8
  paths use documented path findings; oversized canonical handoff specifically
  uses error `handoff.path.too-large`;
- missing, duplicate, invalid, and mismatched canonical markers use their exact
  handoff findings and precedence;
- markers outside Session Identity do not satisfy the check;
- one physical read is reused when the handoff is also a routed path;
- path failure suppresses marker cascades;
- same-packet temporal staleness is explicitly not claimed;
- packet transition removes or replaces the pointer;
- readable INDEX source line passes, while missing/duplicate/literal-path forms
  warn with `index.current-handoff-source`.

### 16.4 Ledger Hygiene

- every v2 open record exact grammar and identity outcome is covered;
- valid markers with empty descriptions and other non-marker grammar defects use
  `ledger.open-item-malformed-record`;
- current-packet open records preserve existing checkpoint behavior;
- previous, missing, and invalid open markers warn non-terminal and error
  terminal;
- malformed open records follow the same conditional severity;
- closed review findings warn non-terminal and error terminal;
- only a current-packet closed TODO during `in_progress` is temporarily quiet;
- every other closed TODO warns non-terminal and errors terminal;
- `Recently Closed`, future/deferred sections, archives, prose, and sentinel text
  remain ignored;
- one line receives no duplicate malformed/identity/closed finding;
- all v1 ledger fixtures and behavior remain unchanged.

### 16.5 CLI And JSON

- standalone `--version` emits exact output and performs no root or state access;
- matching `--require-version` diagnoses normally and mismatch outranks a bad
  root;
- malformed/duplicate requirements use `invalid_invocation`; valid mismatch uses
  `version_mismatch`, all before project access with exit `2`;
- before/after-root option order, unknown arguments, `--`, help precedence,
  split/equals form, duplicate flags, human output, `--json`, and `--strict`
  combinations are deterministic;
- existing v1 JSON calls remain exact schema 1; explicit guarded calls and
  effective-v2 diagnoses use the exact schema-2 completed/error shapes;
- a no-guard failure after effective v2 is established uses schema 2, while a
  pre-version failure retains schema 1;
- schema-2 root and state nullability for preflight, missing, unsupported, and
  effective-v1 lanes is unambiguous;
- no-guard internal failure before versus after effective-v2 establishment
  selects schema 1 versus schema 2 exactly;
- v2-only findings in a v1 diagnosis become internal errors;
- old v3.1 behavior with the new flag is documented as fail-closed invocation.

### 16.6 Agent Experience And Repository Contracts

- fixed startup surface remains at or below 12,000 characters;
- state remains at or below 80 lines;
- current handoff is not an unconditional startup read;
- One-shot and Mini surfaces do not acquire state-v2 requirements;
- the canonical adapter Fast Start says routed membership permits selection but
  does not instruct a full read; current intent, active sections, and targeted
  matches control loading;
- rendered adapters remain in parity with the canonical kernel;
- README image hash remains unchanged;
- README Copy-Paste Start Prompt remains byte-identical to no-clone Option 1;
- their shared expanded prompt teaches the v2 leaf packet, `validation_for`,
  migration preview, and optional current-handoff rules for Standard/Full;
- TODO supplies `## Recently Closed`, and INDEX names state as the sole current
  handoff source;
- repository validator, full unittest discovery, Bash syntax, PowerShell syntax,
  whitespace checks, and Ubuntu/Windows Python 3.10/3.12 CI pass.

## 17. Performance And Maintenance Budget

Measured v3.1 baseline on 2026-07-11:

- fixed startup surface: 9,892 characters;
- state template: 28 lines / 743 bytes;
- handoff template: 3,245 characters;
- minimal-project Doctor process: approximately 133 ms on the audit host;
- full test suite: 230 tests in 35.101 seconds, with 3 environment-dependent
  skips;
- repository validation, prompt synchronization, rendered-agent parity, and
  whitespace checks: clean.

v3.2 acceptance limits:

- no unconditional handoff context load;
- no network, subprocess, Git, or validation-command execution in Doctor;
- no more than one additional bounded handoff read when the pointer is present;
- no parser expansion to arbitrary YAML or nested validation requirements;
- no new dependency outside Python 3.10+ standard library;
- no new persistence for One-shot or Mini;
- no project-specific current-handoff path in INDEX;
- steady-packet overhead is one `validation_for` field and one packet marker per
  active record; transition overhead is the documented coherence transaction
  plus an optional handoff pointer when continuity exists;
- downstream Doctor pins rotate only when Doctor behavior/version changes.

The implementation is expected to touch approximately eight core Python files,
at least five focused test files, agent-experience/repository validators, and the
release-owned templates, examples, onboarding, prompt, documentation, and
metadata surfaces. The compact adapter wording change also regenerates the six
rendered adapters. The realistic release diff is roughly 40-50 files; most of
that surface is contract synchronization and release plumbing, not duplicated
behavior.

The v3.2 implementation does not structurally expand the adapter kernel. It

## 20. Failure Modes And Safeguards

| Failure mode | Safeguard |
|---|---|
| v2 fields leak into v1 interpretation | Version-specific known/required sets and effective-version dispatch |
| Future state version receives old rules | Unsupported-version dependent-check skip |
| Packet ID changes but validation stays old | Required exact `validation_for` plus transition review procedure |
| ID changes while claims stay stale | Fresh claim review; no semantic-proof claim |
| Old handoff contains a matching marker in history | One canonical marker at one fixed Session Identity field |
| Handoff path is invalid and marker also appears missing | Path finding only; marker cascade suppressed |
| Oversized canonical handoff cannot be marker-checked | Dedicated `handoff.path.too-large` error |
| INDEX keeps an old handoff path | Exact source warning plus migration preview |
| Handoff becomes fixed startup context | Optional pointer and intent-routed read rule |
| Checked micro-task blocks every in-progress strict run | Packet/status-aware hygiene matrix |
| Old or unlinked open ledger work survives a switch | Exact v2 record grammar and packet-identity findings |
| Closed review history remains active | Review-specific warning/error and a defined Recently Closed destination |
| Umbrella returns after a child | Procedural leaf review; no Doctor semantic claim |
| Wrong released Doctor runs | Exact pre-diagnosis `--require-version` gate |
| Version guard is described as full provenance | Explicit dirty/fork and reproducibility limitations |
| v1 JSON client breaks | Existing calls keep schema 1; v2/guard selects schema 2 |
| Command-path lint reports false readiness | No arbitrary command parser in v3.2 |
| Platform metadata implies unproven coverage | Portable wrapper and CI guidance instead of state metadata |
| Migration tool blesses guessed semantics | Read-only agent preview with manual semantic decisions |

## 21. Rejected Alternatives

### Natural-Language Objective/Proof Similarity

Rejected because regression checks and task-specific checks can use different
vocabulary, producing false positives. Doctor requires explicit identity and
leaves semantic adequacy to task evidence and review.

### Parent Packet Or Objective Chain In State

Rejected because no canonical registry can verify parent existence or cycles.
The chain would duplicate SPEC authority and increase state-update drift.

### Required Current Handoff

Rejected because small Standard work often needs no handoff. Requiring a 126-line
continuity document would violate small-project compression.

### Default Handoff Read

Rejected because the measured fixed context would exceed its 12,000-character
budget.

### Nested Route Tiers

Rejected because `routed_docs` plus INDEX already separate immediately
selectable candidate routes from on-demand routes. A second taxonomy adds
migration cost without capability.

### Unconditional Report Schema 2

Rejected because current v1 machine clients depend on the exact schema-1 shape.
Schema 2 is selected only by state v2 or explicit use of the new version guard,
so an existing v1 invocation does not break during a minor upgrade.

### Public Ledger Severity Discriminator

Rejected because separate stable ledger IDs and status-only conditional severity
express the required outcomes without expanding the public `Finding` object or
adding hidden linkage metadata.

### Doctor Runtime Lock In v3.2

Rejected because the observed failure is wrong released-version selection, not
same-version runtime tampering. A version guard solves the observed problem with
far less distribution and maintenance work.

### Standalone Doctor Artifact In v3.2

Deferred because it requires a separate build, reproducibility, checksum,
release-asset, installer, and cross-platform launcher design. A later release may
adopt it as a complete distribution feature.

### Command-Path Discovery

Rejected because arbitrary Bash, PowerShell, cmd, package-script, container, and
environment syntax cannot be resolved conservatively from one scalar command.
Executable presence would still not prove runtime capability.

### Platform Or Runner Metadata

Rejected for the core state because Doctor cannot prove coverage from metadata.
Portable wrappers and CI provide a stronger operational contract with less state.

### `sdad migrate --plan` In v3.2

Deferred because the high-value decisions are semantic and the current parser is
not a comment-preserving serializer. The agent migration-preview protocol solves
the observed adoption failure without implying machine certainty.

## Owner-Approved Terminology And User-Control Amendment

This section is an additive correction approved after Tasks 1-7. It supersedes
conflicting state-v2 terminology in earlier sections without rewriting the
completed-task history or changing any released state-v1 behavior.

### Public Identity And One Loop

- Display name: `SDAD Protocol`; the expanded name remains SPEC-Driven AI
  Development and the category is repository-local operating protocol.
- The repository name, slug, remote, raw URL layout, `sdad` CLI name, and Doctor
  name remain unchanged.
- There is one canonical loop: Plan -> Route -> Implement -> Verify -> Report.
  Owner gate and handoff are triggered branches, not additional loop modes.
- Current user vocabulary is scale, execution scope, work packet, validation
  contract, evidence-ready, owner gate, owner acceptance, and handoff. Q1-Q5,
  operating intensity, numeric autonomy, recovery modes, owner checkpoint, and
  AI-complete are internal, legacy, or migration vocabulary only.

SDAD owns repository-local state, routing, authority, bounded evidence, owner
control, decisions, and continuity. It does not prescribe the agent's universal
brainstorming, debugging, planning, review, or implementation method.

### Corrected State-v2 Contract

State v1 remains exact. State v2 is an explicit contract and does not inherit
the state-v1 control list:

```python
V1_STATE_KEYS = (
    "scale", "intensity", "autonomy", "active_spec", "active_packet",
    "owner_gates", "validation", "routed_docs",
)
V2_STATE_KEYS = (
    "scale", "execution_scope", "active_spec", "active_packet",
    "owner_gates", "validation", "routed_docs", "validation_for",
    "current_handoff",
)
```

- `scale`: exactly `standard | full` in state v2.
- `execution_scope`: exactly `unit | packet`; it is required and controls the
  currently authorized work boundary, not risk acceptance.
- `intensity` and `autonomy` are not state-v2 keys. A stale occurrence receives
  the ordinary unknown-key diagnostic and does not activate a v1 autonomy rule.
- `validation_for` remains required. `current_handoff` remains optional.
- State v1 retains its exact scale, intensity, numeric autonomy `0..4`, key,
  message, finding, ordering, severity, human/JSON, and exit contracts.
- Report schema remains 2, and `_select_report_schema()` remains the sole
  state-to-report mapping. No report schema 3 or state schema 3 is introduced.

Existing autonomy-4 `gate.required` behavior runs only for effective state v1.
`execution_scope` never substitutes for an owner gate. The internal finding ID
`gate.q5-review` remains stable, but its state-v2 message, remediation, and
evidence describe a protected-action term/review and contain no Q5 wording.
Keyword inference remains warning-only.

### Routing, Continuity, And Readiness

- `routed_docs` is an eligible selection set. Membership permits a targeted
  read chosen by current intent; it is never a startup read-all instruction.
- New state-v2 bootstrap does not create or route `save-state.md`.
  `current_handoff` is the sole current handoff pointer. A v1 save-state remains
  valid migration input and must not be deleted automatically.
- Keep `docs/work-packet-state.md` for v3.2 and retitle its responsibility
  `Delivery Readiness Model`. It is optional readiness guidance, not current
  packet authority; path renaming remains deferred.
- A handoff links to current authorities rather than duplicating them and is
  loaded only for resume or continuity intent.

### Interaction, Delegation, And Authority

The five kickoff questions become internal inference. Ask at most one blocking
question, and only when the answer changes scale, execution scope, a claim, or
an owner gate. The owner may override the inferred controls.

Every packet or delegated-worker envelope carries outcome/objective,
authority/reference, constraints/scope, validation, evidence/limits,
gates/stop, and report. A worker cannot be assumed to inherit the coordinator's
context or authority.

Guidance, validation, technical enforcement, and owner decision are separate
layers. A Markdown instruction is guidance; Doctor/tests/CI are validation;
permissions or branch protection are enforcement; authorization and acceptance
are owner decisions.

This release does not add required `validation[].limits`, replace the approved
packet-status enum, or add a structured gate framework. Those wire-format
changes require another explicit owner decision.
