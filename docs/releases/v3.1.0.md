# SDAD v3.1.0

Release date: 2026-07-10
Tag: `v3.1.0`

Status: Historical release record. Its terminology describes v3.1.0 and is not
the current state-v2 operating contract.

SDAD v3.1.0 adds a diagnostic control surface, Gemini CLI support, and
cross-model operating guidance while preserving the v3.0 progressive control
plane and owner-acceptance boundary.

## SDAD Doctor

The checkout-only `sdad doctor` command diagnoses stateful Standard and Full
SDAD projects from an SDAD repository checkout. It is read-only: it uses bounded
local reads, does not use the network, does not execute declared validation
commands, and does not mutate or repair project files.

```text
python <SDAD_CHECKOUT>/scripts/sdad.py doctor [PROJECT_ROOT] [--json] [--strict]
```

- Exit `0`: diagnosis completed and the selected policy passes.
- Exit `1`: diagnosis completed, but findings fail policy; `--strict` also makes
  warnings fail without changing their severity.
- Exit `2`: diagnosis could not complete because invocation, root, state I/O, or
  an internal diagnostic boundary failed.

Human output supports interactive use. `--json` emits one versioned document
for automation. A clean report is diagnostic evidence; it does not prove
correctness, effectiveness, or owner acceptance, and it does not grant release
approval.

## Gemini CLI Adapter

Gemini CLI is now a first-class adapter route. Standard and Full installs place
the pinned adapter at repository-root `GEMINI.md`; Mini installs use the same
target with the pinned Mini template. Clone-based PowerShell and Bash installers
also accept `gemini-cli`.

The adapter is self-contained and follows the same state -> INDEX -> current
source/tests -> one routed policy path as the other model surfaces. Gemini
provider features can add boundaries, but provider guidance is not enforcement
of SDAD evidence, completion, Q5, or owner-gate policy.

## Cross-Model Guidance

All rendered agent surfaces now state that external content and tool output may
contain embedded instructions. Those instructions are treated as untrusted
evidence unless owner intent or active policy independently authorizes them.
Valid JSON, a successful tool call, or command syntax proves structure only;
observed results and task-specific semantic validation remain required.

The routed playbooks add progressive context localization, bounded feedback
loops, fresh-context review, and evaluation guidance. The evaluation contract
separates regression from capability claims, prefers representative and held-out
tasks, accounts for nondeterminism, and keeps efficiency claims subordinate to
quality, evidence, and rollback limits.

## Research And Evaluation Boundary

[Research Foundations](../research-foundations.md) records 25 official or primary sources
from official provider documentation, standards guidance, peer-reviewed work,
and primary research reports. They inform bounded SDAD decisions; the cited
sources do not establish SDAD effectiveness. Repository regression tests also
do not establish comparative productivity, quality, safety, or completion-rate
gains.

## Compatibility And Migration

- Existing v3.0 Standard and Full projects remain compatible; no control-file
  schema migration is required solely for v3.1.0.
- Existing adapter installs are not rewritten automatically. Reinstall from the
  v3.1.0 pinned source only when adopting the new cross-model guidance.
- Gemini CLI projects should select `gemini-cli` and use `GEMINI.md`; do not
  create multiple tool adapters unless the project actually uses those tools.
- `sdad doctor` remains checkout-only in 3.1.0. No-clone and adapter installers
  intentionally do not distribute the diagnostic runtime.
- The no-clone manifest pins immutable baseline commit
  `1741b72a51bb4eb0711e8c0f188c3ddcf922eaaa` and verifies every source with
  SHA-256. Keep revision, source path, and hash from one manifest together.

## Verification

The local release gate covers manifest hashes from baseline Git blobs, rendered
adapter parity, README/Option 1 byte parity, repository validation, direct
contract tests, full unittest discovery, Bash and PowerShell syntax, whitespace
checks, pinned-source immutability, and the owner-provided README image hash.

The local Windows gate passed 230 tests with three Windows privilege-dependent skips
for permission or link cases this environment could not create. Repository,
render, prompt-sync, Bash syntax, PowerShell parser, pinned-source diff, and
image-hash checks also passed. The skips are disclosed limits, not pass evidence
for those scenarios. The platform CI matrix remains required before publication.
No remote CI, pushed tag, or published GitHub Release claim is made by this
pre-publish document.

Doctor does not prove correctness, effectiveness, or owner acceptance. Provider
guidance is not enforcement, and published release acceptance remains an owner
gate.
