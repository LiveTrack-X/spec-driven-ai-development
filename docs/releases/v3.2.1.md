# SDAD v3.2.1

Release date: 2026-07-14
Tag: `v3.2.1`

SDAD v3.2.1 is a bounded documentation and packaging correction. It clarifies
what the protocol governs, keeps implementation methods open, and removes
internal Superpowers coordination artifacts from the distributed tree. It does
not introduce a new state or report schema.

## Positioning And Distribution Cleanup

The expanded public name is now SPEC-Directed AI Development. The stable
descriptor is "a repository-local operating protocol for AI-assisted
development." SDAD remains method-agnostic, tool-neutral, and model-neutral: it
does not prescribe TDD or another implementation method, run or schedule
agents, or act as an AI coding harness or orchestrator.

Repository-local scope, validation, evidence, unresolved state, owner gates,
and owner acceptance remain the protocol boundary. Internal `.superpowers/`
task reports and `docs/superpowers/` implementation plans are no longer part of
the distributed tree, and both paths are root-ignored for future work.
The confirmed SDAD 3.0 README artwork remains unchanged; its embedded
`Spec-Driven` label is historical rather than the current expansion.

## State v2 And Packet Identity

State schema v2 is unchanged. Stateful Standard and Full installations retain
one executable leaf `active_packet`, exact `execution_scope: unit | packet`,
and `validation_for` equal to the active packet ID. State schema versions remain
separate from Doctor and report schema versions.

## SDAD Doctor 3.2

The checkout-only, read-only Doctor reports version 3.2.1 and supports the
exact guard:

```text
python <SDAD_CHECKOUT>/scripts/sdad.py doctor [PROJECT_ROOT] --require-version 3.2.1 [--json] [--strict]
```

Existing unguarded state-v1 JSON calls retain report schema 1; guarded or
state-v2 calls use report schema 2. No finding ID, severity, check order, state
schema, or report schema changed in this patch.

## Handoff, INDEX, And Ledger Consistency

The v3.2 consistency contract is unchanged. Doctor checks the packet identity
across state, `validation_for`, routing INDEX, active TODO and review ledgers,
and an optional `current_handoff`.

## Targeted Routing

The startup route remains adapter -> `sdad-state.yaml` -> `docs/INDEX.md`, then
only the route needed for the current intent. `routed_docs` remains an eligible selection set rather than a startup read-all order.

## Migration Preview And Upgrade Compatibility

The read-only preview and state-v1 compatibility contract are unchanged.
Existing installations are not modified automatically. Reinstall an adapter or
skill only when adopting the 3.2.1 wording. No-clone sources pin the immutable
baseline `f173aa398562d6a9d86b941dc79f75f9381148f4`; keep its revision, source
path, and SHA-256 together.

## Verification Evidence

The local release gate covers the seven pinned Git-blob hashes, manifest and
installer parity, rendered agent surfaces, Copy-Paste Start Prompt parity,
repository validation, unittest discovery, shell syntax, and whitespace.
The local Windows release gate ran 392 tests, with three environment-dependent
skips.

The existing fresh-context task benchmark remains the evidence boundary for
context recovery. This patch is not a controlled comparison and does not claim
improved productivity or general effectiveness.

## Known Limitations

Doctor green proves structural consistency only. It does not prove correctness, productivity, or owner acceptance.
SDAD does not execute declared validation,
replace tests or CI, enforce permissions, or guarantee that an evidence claim
is semantically sufficient. Tool policy, review, branch protection, and owner
judgment remain separate controls.
