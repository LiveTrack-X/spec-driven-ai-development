# SDAD v3.2.0

Release date: 2026-07-12
Tag: `v3.2.0`

SDAD v3.2.0 strengthens the repository-local protocol around one active work
packet, packet-owned validation, targeted context, and explicit owner control.
The public framework name is SDAD Protocol; the repository slug and install
paths remain unchanged.

## State v2 And Packet Identity

New stateful Standard and Full installations use state schema v2. The state has
one executable leaf `active_packet`, exact `execution_scope: unit | packet`, and
`validation_for` equal to the active packet ID. State v2 removes numeric
autonomy and operating intensity from the current contract. Scale determines
the persistent control surface; execution scope determines the current work
boundary; owner gates remain separate protected-action authority.

Existing state-v1 inputs remain accepted under their original compatibility
contract. State v2 is now released and its meaning is frozen: a future semantic
replacement requires a new state schema version.

## SDAD Doctor 3.2

The checkout-only, read-only Doctor reports version 3.2.0 and supports an exact
guard:

```text
python <SDAD_CHECKOUT>/scripts/sdad.py doctor [PROJECT_ROOT] --require-version 3.2.0 [--json] [--strict]
```

Doctor version, state schema version, and report schema version are separate
domains. Existing unguarded state-v1 JSON calls retain report schema 1;
guarded or state-v2 calls use report schema 2. Doctor does not execute declared
validation commands, modify project files, or grant release authority.

## Handoff, INDEX, And Ledger Consistency

For state v2, Doctor checks the exact packet identity across the state,
`validation_for`, routing INDEX, active TODO and review ledgers, and an optional
`current_handoff`. A handoff must be readable, bounded, packet-marked, and named
by the current state and INDEX source. Legacy save-state remains migration
history rather than a second v2 continuity pointer.

## Targeted Routing

The startup route is adapter -> `sdad-state.yaml` -> `docs/INDEX.md`, followed
by current source/tests and only the route needed for the intent. `routed_docs`
is an eligible selection set, not a startup read-all order. Actual documents
read belong in the evidence report.

Agent surfaces and the global Codex skill infer scale and controls first, ask at
most one blocking question when the unresolved fact changes scale or an owner
gate, and preview state-v2 bootstrap or migration changes before writes.

## Migration Preview And Upgrade Compatibility

The skill produces a read-only preview before migrating an existing stateful
project. It identifies v1-to-v2 mappings, files to add or change, validation
ownership, optional handoff effects, and preserved legacy inputs. It does not
rewrite the project until the owner accepts that migration scope.

Existing v3.1 projects are not modified automatically. Reinstall the adapter or
global skill only when adopting the v3.2 instructions. No-clone sources pin the
immutable behavior baseline
`b433b4cbf490bd875a40b76127abefbefed3f243`; keep revision, source path, and
SHA-256 from one manifest together.

## Verification Evidence

The release gate covers exact seven-source Git-blob hashes, manifest/source
parity, rendered agent surfaces, Copy-Paste Start Prompt parity, canonical
README artwork, repository validation, full unittest discovery, Bash and
PowerShell syntax, whitespace checks, clean-tree replay, and immutable baseline
ancestry.

A bounded fresh-context task benchmark recovered the active packet, next
action, validation and claim boundary, owner gate, and intended document routes
from adapter -> state -> INDEX. That evidence supports only this task-scoped
context-recovery claim. It is not a controlled comparison with v3.1 and does
not establish productivity or general effectiveness.

The local Windows gate ran 392 tests, with three privilege-dependent skips.
Repository validation, rendered-surface, prompt-sync, Bash syntax, PowerShell
parser, immutable-source, artwork-hash, and whitespace checks also passed.
Platform CI remains required before publication.

## Known Limitations

Doctor green proves structural consistency only. It does not prove product
behavior. A clean report does not prove correctness, productivity, or owner acceptance.
The Doctor does not determine whether a
declared command semantically proves its prose claim, execute validation, or
replace project-specific review. Markdown records authority but does not
technically enforce permissions; tool policy, hooks, sandboxing, and branch
protection remain separate enforcement mechanisms.

Tool-native sessions, checkpoints, memory, and diagnostics are conveniences,
not SDAD state, handoff, or Doctor authority. The three Windows
privilege-dependent permission/link cases may remain skipped where the local
environment cannot create those conditions; CI and real platform checks bound
that gap.
